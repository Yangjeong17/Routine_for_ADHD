## 분석: 어떻게 direction을 줬으면 더 효율적이었을까

전체 세션을 보면 오류나 비효율의 원인이 크게 4가지 패턴으로 나뉩니다.

---

### 1. 가장 큰 문제: "이전 세션 결과 확인 없이" 다음 요청

#### 실제로 발생한 일:

- Session 2에서 `TimelineBlock.tsx` dead code 문제가 발견됐는데, Session 3·4·5에서도 계속 같은 문제가 반복됐습니다. 이슈 로그에 `[REVIEW_REQUIRED]`로 3번 이월됐고, 결국 Session 5에서야 해결됐습니다.
- Session 4에서 `CurrentTimeIndicator` 미통합 문제가 동일하게 3회 이월됐습니다.

왜 이렇게 됐냐면: 저는 새 세션마다 context를 새로 시작합니다. 이전 세션 이슈 로그를 참조하지 않으면 무엇이 미해결인지 모릅니다.

#### 개선 방법:

```

#### 세션 시작할 때 이렇게 해주세요

@.ai-docs/issues/IssueLog_Session_4_... 읽고
미해결 항목부터 처리한 다음에 step5 작업 시작해줘.

```

새 작업 지시 전에 "미해결 이슈 파일을 먼저 @참조해서 읽고 시작해" 라는 한 줄을 붙이면 됩니다.

---

### 2. 요청 방식: 여러 변경사항을 한 메시지에 섞어서 전달

#### **실제로 발생한 일:**

`step6_detailed.md`와 `전체 수정.md`는 각각 6~8개의 서로 다른 변경 사항이 섞여 있었고, 일부는 이전 작업을 **되돌리는 내용**(BlockPopover 제거 → 재도입 → 다시 제거)이었습니다. 

그 결과:

- BlockPopover가 제거됐다가 다시 생겼다가, 이번에 또 제거했습니다.
- 전체 수정.md의 코드 스니펫에 import 문이 없어 빌드 에러가 134개 발생했습니다.

#### **개선 방법:** 한 번에 하나의 주제로 요청하거나, 우선순위를 명시해주세요.

```

#### 나쁜 방식

step6에 있는 거 다 해줘.

#### 좋은 방식

step6에서 이것만 먼저 해줘:

1. BlockPopover 제거 (완료되면 확인 후 다음 진행)
2. 카테고리 버튼 선택 UI
```

또는 전체 수정.md처럼 코드를 직접 제공할 때는:

```

아래 코드로 파일을 교체해줘. import 문은 기존 것을 유지하고 내용만 바꿔줘.

```

라고 명시하면 import 누락 문제를 막을 수 있습니다.

---

### 3. 변경 후 검증 요청이 없었음

#### 실제로 발생한 일:

- `전체 수정.md` 파일을 직접 수정하고 "수정된 파일 확인해봐"라고 했는데, 이미 코드가 적용된 상태였지만 import 없이 저장된 상태였습니다.
- 여러 세션에서 빌드를 직접 실행해보지 않고 넘어간 경우가 있었고(Session 5 로그의 "빌드 최종 확인 미완"), 다음 세션에서 그 에러가 터졌습니다.

#### **개선 방법**: 파일 수정 후에는 항상 검증 요청을 붙여주세요.

```

수정한 다음에 npm run build 돌려서 에러 없는지 확인해줘.

```

저도 기본적으로 빌드를 해야 하지만, 명시적으로 요청하면 절대 빠뜨리지 않습니다.

---

### 4. 방향 전환 시 "이전 결정과 충돌"을 알려주지 않음

#### 실제로 발생한 일:

- Step 6 세션에서 "BlockPopover 제거"를 작업했고, 로그에도 "완료"로 기록됐습니다.
- 이후 `전체 수정.md`에서 BlockPopover를 다시 살리는 작업이 왔는데, 이전 결정과 충돌한다는 언급이 없었습니다.
- 결과적으로 이번에 또 제거했습니다. 총 3번의 작업이 낭비됐습니다.

#### **개선 방법:** 이전에 한 결정을 뒤집을 때는 명시해주세요.

```

#### 이렇게 해주세요

이전에 BlockPopover 제거했는데, 다시 살리고 싶어.
카드 클릭 → 팝오버, 펜 아이콘 → 모달로 바꿔줘.

```

### 요약: 세션 시작 템플릿 제안

이런 형태로 세션을 시작하면 오류가 크게 줄어들 것입니다.

```

## 이번 세션 작업

### 사전 확인

@.ai-docs/issues/IssueLog_Session_X_... 읽고 미해결 항목 있으면 먼저 처리해줘.

### 이번에 할 것 (우선순위 순)

1. [기능명]: [한 줄 설명]
2. [기능명]: [한 줄 설명]

### 이전 결정과 달라지는 것

- [변경된 결정 명시]

### 완료 조건

- npm run build 성공
- 브라우저에서 [확인할 동작] 작동 확인
```

이 중에서 실질적으로 가장 효과가 큰 것 하나만 꼽으면 
**"이전 이슈 파일을 @참조해서 세션을 시작하는 것"**입니다. 
dead code 문제 하나가 4개 세션에 걸쳐 반복됐던 게 그 증거입니다.
